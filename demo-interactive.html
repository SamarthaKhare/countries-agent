Architecture Overview

A three-node LangGraph StateGraph orchestrates the full query lifecycle — intent parsing, API retrieval, and answer synthesis — each concern isolated in its own node and wired together with explicit conditional error routing.

FastAPI  ·  POST /api/query  ·  GET /api/health  ·  Swagger /docs
① IDENTIFY

Sends the raw query to the LLM and returns structured JSON: country name, requested fields, and a validity flag.

② FETCH

No LLM involved. Calls restcountries.com, normalises the raw schema, handles 404 / 5xx errors.

③ SYNTHESIZE

Grounds the answer entirely in the API data — no hallucination possible. Produces a natural-language response.

Limitations & Trade-offs

Honest assessment of what the current architecture does and doesn't handle well.

⚠ Known Limitations

  • Stateless — no conversation memory across turns
  • Country disambiguation delegated to API fuzzy search ("Congo" → first match)
  • Groq free tier: 14,400 req/day rate limit
  • IDENTIFY node is synchronous — blocks the async event loop
  • No response caching — identical queries always hit the LLM

✓ Design Trade-offs

  • LangGraph separates concerns — each node is independently testable
  • OpenAI SDK for all providers — swap with one env var
  • API normalised once in tools.py — downstream code never sees raw schema
  • Conditional edges make error routing explicit, not implicit
  • Groq default: free tier + sub-second inference
Provider flexibility

Set LLM_PROVIDER=groq|gemini|anthropic to switch models. All three providers use the OpenAI-compatible REST endpoint via the openai SDK — only the base_url, api_key, and model string change.
